# MCP Security Platform — Critical Review

**Date:** 2026-05-16
**Reviewer:** AI-assisted deep review (appsec + claimed-vs-actual audit), all findings verified against source at cited file:line.
**Verdict:** **BLOCKED for any shared/production environment.** 2 CRITICAL + 4 HIGH security findings, plus a stale "single source of truth" architecture that omits the platform's largest current subsystem.

This document supersedes `docs/appsec-review.md` (dated 2026-04-21, which pre-dates the credential broker, OAuth router, Vault, and migrations V006–V008 and is now stale).

---

## 0. TL;DR — what is actually true

This is **not vaporware**. Layer 2 (proxy: ~6,400 LOC), Layer 1 (gateway config), and Layer 3 (audit logger + compliance checker) are substantively real and internally consistent. The problems are:

1. **Two CRITICAL crypto/identity failures** in the new, undocumented credential broker (CB-001 identity collapse, CB-002 plaintext master-key transport).
2. **Two prior HIGH findings (F-001, F-002) remain OPEN.**
3. **The "canonical" `ARCHITECTURE.md` (v1.0.0) does not mention the credential broker, Vault, `credential_store`, or the OAuth router at all** — the largest recent subsystem is invisible in the doc that calls itself the single source of truth.
4. **Documentation hallucinations**: SPDX SBOM, outbound Jira, Helm/K8s readiness, "per-tool" rate limiting, and unsourced marketing stats are described as real but are absent or empty in code.

## 1. Security Findings (severity-ranked)

| ID | Sev | Component | File:Line | Issue |
|---|---|---|---|---|
| **CB-001** | **CRITICAL** | OAuth enrollment | `proxy/app/routers/oauth.py:73-74,86-87`; `gateway/nginx/conf.d/mcp-proxy.conf:132` | `user_sub` (KEK derivation + credential_store row key) is read from client-controllable `X-Client-Cert-CN`, defaulting to literal `"unknown"`. No `/auth/` nginx route exists (routes return 404). All enrolled credentials collapse to one identity and one KEK = `HMAC(master, "unknown")`. `ON CONFLICT DO UPDATE` lets any enroller overwrite any other user's stored refresh token. |
| **CB-002** | **CRITICAL** | Vault KMS | `proxy/app/core/config.py:210-211`; `proxy/app/credential_broker/kms.py:29` | `VAULT_ADDR` defaults to `http://` — the master secret protecting **all** encrypted credentials transits the network in cleartext. `VAULT_TOKEN` weak default; only blocked when `ENVIRONMENT=production` (lab/dev unprotected). |
| **F-001** | **RESOLVED** | Proxy network trust | `docker-compose.yml` networks; `scripts/check_network_isolation.py` | App-layer (no header trust) + nginx CN-blanking already in. Now also: proxy removed from the flat `internal-net` mesh and from `observability-net`; egress via dedicated pairwise `proxy-{opa,ollama,redis,db}-net` + `vault-net`; inbound only via `gateway-net`. compliance-checker can no longer reach proxy. Regression-gated in `make security-check`. Residual TCP-bidirectional reach is now one compromised backend (not the whole mesh), neutralised by the app-layer identity fix. |
| **F-002** | **HIGH (OPEN)** | OPA bundle signing | `docker-compose.yml:209-220` | OPA command has no `--config-file`/`--set=bundles.*.signing.*`. Bundle signature verification is not enforced in any environment. INV-012 is documentary only. **Not closed.** |
| **CB-003** | **HIGH** | OAuth CSRF state | `proxy/app/routers/oauth.py:57-65,73-88` | State is `HMAC(secret, "{client-supplied X-Session-Id}:{user_sub}")` with no server-side nonce. Attacker controlling both headers (or knowing `OAUTH_STATE_SECRET`) forges valid state. No replay protection. |
| **CB-004** | **HIGH** | INV-001 violation | `proxy/app/routers/oauth.py:81-116` | Credential enrollment writes to `credential_store` and emits **no audit event**. Credential lifecycle is invisible to compliance/anomaly. |
| **CB-005** | **HIGH** | INV-011 / availability | `infra/db/migrations/V008__role_assignments.sql` (CREATE TABLE, **no GRANT**); `proxy/app/middleware/auth.py` `_load_roles()` | `proxy_app` has no privilege on `role_assignments`. The exception is swallowed → every authenticated client resolves to **zero roles** → every protected endpoint 403 in a fresh non-lab deploy. Verified: V008 contains no GRANT. |
| CB-006 | MEDIUM | Lab secrets | `.env.lab` | Contains functional secrets (`DEX_CLIENT_SECRET`, `VAULT_TOKEN=lab-root-token`, all HMAC keys = `devpassword`, `OAUTH_STATE_SECRET`). **Verified `.env.lab` was never committed to git** (history clean) — so this is a weak-default/operator-hygiene issue, not a leaked-secret incident. The inline "rotate them" comment is precautionary. |
| CB-007 | MEDIUM | KDF | `proxy/app/credential_broker/approaches/approach_a.py:12-14` | KEK = single-round unsalted `HMAC(master, user_sub)`. Use HKDF (RFC 5869). If master leaks, all KEKs are one HMAC away. |
| CB-008 | MEDIUM | Key lifetime | `proxy/app/credential_broker/broker.py:39-47` | `_master_secret` cached for process lifetime, never zeroed, no rotation/TTL re-fetch. Heap dump = permanent master compromise. |
| CB-010 | MEDIUM | INV-002 | `adapters/m365.py:71-75`, `adapters/bitbucket.py:58-65`, `adapters/dex.py` | `resp.raise_for_status()` lets IdP error body (may contain token/secret echoes) propagate uncaught to the default 500 handler / logs; not covered by redaction patterns. |
| CB-011 | MEDIUM | OAuth | `adapters/{m365,bitbucket,dex}.py`, `oauth.py` | No PKCE on any authorization-code flow. Dex supports `code_challenge`; intercepted codes are exchangeable without proof of possession. |
| CB-012 | MEDIUM | INV-007 spirit | `infra/db/migrations/V006__credential_store.sql`; `broker.py:93-124` | `DELETE` granted on `credential_store` with no audit-before-delete enforcement; conflicts with WORM/audit philosophy for security-sensitive records. |
| CB-009 | MEDIUM | TLS | `proxy/app/credential_broker/kms.py:28-33` | Vault `httpx.AsyncClient` has no explicit `verify=`; default + `http://` default (CB-002) → token in cleartext. |
| CB-015 | LOW | Config drift | `.env.example` | Zero broker variables documented; operators using `.env.example` inherit `config.py` hardcoded defaults (`VAULT_TOKEN="change-me-in-production"`). |

**The single most dangerous issue: CB-001.** It collapses every brokered third-party credential (M365, Bitbucket, Grafana, Netbox, Dex) onto one identity and one derivable key, and lets any enroller overwrite any other user's tokens. It is a complete failure of the credential broker's security model.
