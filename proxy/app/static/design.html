The Thesis

Mediate. Don't classify.

Static scanning can't catch semantic capability. A server that wraps a C2 framework looks like any other server — until it calls home.

01

Identity at the edge

Nginx terminates mTLS and injects the verified client CN. Every downstream service inherits a signed identity — not a header it made up.

02

Policy, not heuristics

OPA/Rego evaluates every invocation. Deny-by-default unless inventory says allowed. Policy as code, testable, versionable, auditable.

03

Audit you can trust

SHA-256 hash/event, redaction before storage, Loki-shipped and MinIO-archived. Compliance checker runs daily. No silent gaps.

Credential Broker

Five injection modes

Each tool in the registry declares one mode. The dispatcher reads it at call-time, decrypts, and injects the right header — the MCP server never sees a raw secret.

none

No credential

Upstream is called without any injected auth. Use for public endpoints, sandboxed tools, or where the MCP server manages its own auth.

Proxy ──────────────────→ MCP server (no header added)
no secrets stored default mode
service

Shared service credential

One API key or client secret shared across all users of this tool. Admin uploads it once. Stored encrypted, keyed as __service__. All calls get the same header.

Admin uploads API key credential_store: user_sub=__service__ AES-256-GCM(HKDF-SHA256 KEK) On call: decrypt → Authorization: Bearer <key>
AES-256-GCM HKDF-SHA256 KEK admin-managed
user

Per-user personal credential

Each user uploads their own token (personal API key, PAT, etc). Stored separately per user_sub. User A's call never touches User B's credential.

User uploads own token (user_sub = a3f8c1d2-7b4e-4f12) credential_store: encrypted per user_sub On call: sub from session → decrypt(user_sub) → Authorization: Bearer <user-tok>
per-user isolation user self-service HKDF KEK per sub
service_account

Keycloak client_credentials

Proxy fetches a short-lived token from Keycloak using the tool's registered kc_client_id + client secret. Token is not stored between calls — fetched fresh each time.

Admin stores KC client_secret in credential_store user_sub=__kc_sa__, service=kc_client_id On call: POST /token (client_credentials) Keycloak → access_token (TTL ~5min) → Authorization: Bearer <kc-token>
client_credentials flow short-lived tokens Keycloak-issued
oauth_user_token

User token exchange

The user's Keycloak access token is exchanged for an audience-scoped token targeting the upstream service. No separate secret stored — the user's session is the credential.

User's KC access token (from session) + kc_token_audience on tool record POST /token (token_exchange) subject_token=<user-kc-tok> audience=<upstream-service> → Authorization: Bearer <exchanged-tok>
token exchange (RFC 8693) no secrets stored audience-scoped
Secret Lifecycle

Upload → Encrypt → Inject

A secret never travels in plaintext after it crosses the admin upload boundary. Here's every step.

ADM
Step 1 — Upload

Admin submits plaintext via HTTPS

PUT /admin/credentials/{tool_id} with { secret, credential_type, owner_type, user_sub? }. TLS in transit. Requires admin role. Audit event emitted synchronously before the response returns.

KEK
Step 2 — Key derivation

HKDF-SHA256 derives a per-identity KEK

The broker derives a Key Encryption Key from the master secret using HKDF-SHA256(master, info="mcp-credential-broker-kek-v1:{user_sub}"). Service credentials use __service__ as the identity. This means the master secret alone cannot decrypt any credential — the identity must also be known.

ENC
Step 3 — Encryption

AES-256-GCM with 96-bit random nonce

Plaintext encrypted as nonce(12B) ‖ ciphertext+GCM-tag. A fresh random nonce per write prevents ciphertext reuse. The GCM tag detects any tamper — decryption throws InvalidTag if the user_sub is wrong or the blob is modified.

DB
Step 4 — Storage

Encrypted blob stored in PostgreSQL

Only the encrypted blob lands in credential_store. Row keyed by (user_sub, service, tool_id). Plaintext never written to disk, never logged. Vault holds the master secret, never the per-credential key.

DIS
Step 5 — Dispatch (on tool call)

Dispatcher reads injection_mode from tool_registry

On every tool invocation, the dispatcher reads injection_mode from the tool's row in tool_registry, routes to the correct handler, and fetches + decrypts the credential scoped to the calling user's sub.

DEC
Step 6 — Decryption (in-memory only)

Plaintext exists only for the duration of the request

The broker decrypts the blob in memory. The CredentialResult object has a .zero() method that overwrites the token value in-place. Plaintext never written anywhere after this point.

INJ
Step 7 — Header injection

Upstream call receives a single HTTP header

The dispatcher returns { inject_header: "{inject_prefix} {token}" } — for example Authorization: Bearer eyJ… or X-Api-Key: sk-…. This dict is merged into the upstream request. The MCP server never sees the credential store, Vault, or Keycloak.

Secrets Per MCP

Two actors. Two flows.

Service owners set a shared credential once. Users manage their own tokens independently. Neither can read the other's secrets — the crypto enforces it.

Service Owner / Admin

Shared service secret

One secret covers all callers of this MCP. Upload once — every user's request gets the same injected header. Rotation happens here too.

credential_store user_sub = __service__ ← shared sentinel tool_id = echo-ping encrypted = AES-256-GCM( HKDF(master, __service__) ) ────────────────────────────────── All users of echo-ping get this credential. No per-user row required.
⚠ Audit event emitted synchronously

End User

Personal token per MCP

Each user stores their own PAT, API key, or OAuth token. Stored under their Keycloak user_sub. Alice's secret is invisible to Bob — the KEK is derived from the identity.

credential_store user_sub = a3f8c1d2-7b4e-4f12 · (keycloak sub UUID) tool_id = jira-search encrypted = AES-256-GCM( HKDF(master, user-sub-uuid) ) ────────────────────────────────── Bob's row uses HKDF(master, bob-sub). Decrypting Alice's blob with Bob's KEK → InvalidTag.
keyed to your identity no admin access needed revocable by you

Save My Token ↑
Crypto isolation guarantee

HKDF-SHA256 derives a unique Key Encryption Key per identity. The master secret + the identity sub are both required to decrypt. Users cannot read each other's secrets. Platform admins with Vault access can derive any KEK — Vault audit log + least-privilege are the operational controls.

Injection at call time

At each MCP invocation the proxy reads injection_mode from the tool registry, decrypts the right credential (service or per-user), injects the header, then zeros the plaintext in memory.

Self-service MCP

Users can also manage their MCP access profiles via the self-service-mcp (port 8108) — a JSON-only MCP server with 7 tools: list_available_mcps, enable_mcp, disable_mcp, get_profile, list_functions, enable_function, disable_function. No bloat, no HTML.

Architecture

Nine Layers

Each layer owns a discrete security concern. No layer trusts the one above it.

LAYER 01
API Gateway
Nginx with mTLS termination, rate-limiting per client CN and IP, certificate CN header injection. Only entry point for external traffic.
● enforced
LAYER 02
Policy Engine (OPA)
Rego policies — deny-by-default authz (INV-003), anomaly scoring, and static tool analysis. Evaluated on the REST invocation path.
● enforced
LAYER 03
Security Proxy
FastAPI. Core pipeline: auth → RBAC → quarantine → OPA → credential injection. OIDC/PKCE S256, Keycloak, session JWT. CycloneDX SBOM per tool.
● enforced
LAYER 04
Credential Broker
Two approaches — Approach A (AES-256-GCM/HKDF, PostgreSQL refresh-token store) and Approach B (OAuth provision with Redis token cache). Wired for service/user modes; service_account and oauth_user_token modes not yet implemented in the invocation pipeline.
◐ service/user wired
LAYER 05
Business Services
Invocation pipeline, tool manifest auditor (Ollama, advisory), anomaly detection (sliding-window heuristics), SBOM service, compliance checker.
● enforced
LAYER 06
Observability & Audit
mcp-audit-logger package: Loki emitter, redaction engine, audit event schema. SHA-256 hash/event. MinIO archival. Daily compliance checks (advisory).
● enforced
LAYER 07
Data & Persistence
PostgreSQL 16 with 20 applied migrations (V001–V020). Redis for role and anomaly caching. MinIO WORM for long-term audit archival. Vault for broker key material.
● enforced
LAYER 08
Test Suite
440 tests across unit, integration, security (AI prompt-injection + sandbox escape), and performance. F-001 isolation gated in make security-check.
● 440 tests
LAYER 09
DevOps & Infrastructure
Four Compose stacks (prod, dev, lab, opa-signed). Rootless Podman. Gitea, Keycloak, Dex, NetBox in lab. Helm stubs. step-ca internal CA. Ship-check gate pre-publish.
● lab proven
LAYER 10
Self-Service MCP / Profile Engine
FastMCP server (port 8108) exposing 7 tools for per-identity MCP permission management: list_available_mcps, enable_mcp, disable_mcp, get_profile, list_functions, enable_function, disable_function. Writes to mcp_profiles + append-only mcp_profile_events audit table. Identity from X-User-Sub (proxy-injected); admin manages any profile, agents manage only their own.
◐ OPA integration roadmap
